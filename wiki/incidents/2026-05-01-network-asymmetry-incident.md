# Incident Report: Network Asymmetry & Service Discovery Failure (2026-05-01)

## Executive Summary
During the maintenance period of node `talos-cp-02`, the Kubernetes cluster experienced significant network instability and observability degradation. Clients (specifically Google Chrome on macOS) encountered intermittent `ERR_CONNECTION_REFUSED` errors when accessing internal services via the Traefik LoadBalancer VIP (`10.10.20.56`). Simultaneously, VictoriaMetrics (`vmagent`) reported a complete failure in target discovery (`ScrapePoolHasNoTargets`).

## 1. Root Cause Analysis (RCA)

### 1.1 Network Asymmetry (TCP Reset Phenomenon)
- **Topological Bottleneck**: Traefik was deployed as a `Deployment` with 2 replicas, both residing on node `talos-cp-01`.
- **L2 Failover Conflict**: MetalLB (Layer 2 mode) elected node `talos-cp-03` as the leader for the VIP.
- **Asymmetric Routing Path**: 
    1. Client (VLAN 20) -> VIP (Node `cp-03`).
    2. Node `cp-03` -> VXLAN Tunnel -> Node `cp-01` (Traefik Pod).
    3. Traefik Pod -> Egress directly to Gateway (OPNsense) or Client.
- **Firewall Rejection**: OPNsense, acting as a stateful firewall, received TCP ACK/PSH packets from `cp-01` for a session that technically started on `cp-03` (MAC mismatch). This triggered a security violation, resulting in OPNsense sending a **TCP RST** to the client, translated by Chrome as `ERR_CONNECTION_REFUSED`.
- **Browser Behavior**: Chrome's socket pooling kept "zombie" connections alive; a process restart temporarily fixed the issue by forcing a fresh SYN, which eventually desynchronized again.

### 1.2 Service Discovery Collapse (VictoriaMetrics)
- **KubePrism Timeout**: Talos OS uses KubePrism (local API proxy at `127.0.0.1:7445`). `vmagent` was attempting to "watch" the API via KubePrism.
- **Dead Member Hang**: KubePrism continued to route a portion of requests to the offline `talos-cp-02`. Because `cp-02` was not formally removed from the etcd quorum, KubePrism waited for health-check timeouts (up to 35s), causing the `vmagent` long-polling watch to hang and eventually drop all targets from its cache.

## 2. Implementation of Solution (Option A: Distributed & Symmetric)

### 2.1 Traefik DaemonSet Transition
- **Action**: Modified `traefik-values.yaml` to change `deployment.kind` from `Deployment` to `DaemonSet`.
- **Result**: Every active node (`cp-01`, `cp-03`) now hosts a Traefik pod, ensuring a local entry point is always available.

### 2.2 ExternalTrafficPolicy Enforcement
- **Action**: Set `service.spec.externalTrafficPolicy` to `Local`.
- **Result**: MetalLB only advertises the VIP from nodes where a Traefik pod is healthy. More importantly, it forces the kernel to process the traffic locally. This eliminates inter-node tunneling and ensures **Symmetric Routing**, satisfying OPNsense stateful inspection.

### 2.3 etcd Quorum Stabilization
- **Action**: Formally removed `talos-cp-02` from the etcd cluster using `talosctl etcd remove-member`.
- **Result**: KubePrism immediately updated its routing table, pointing only to the 2 active nodes. `vmagent` discovery resumed within seconds.

## 4. Recommendations
- **Maintain DaemonSet Topology**: Keep Ingress Controllers as DaemonSets to ensure local traffic processing.
- **BGP Migration (Future)**: Consider moving from MetalLB L2 to BGP to eliminate ARP-based failover delays and dependencies.
- **Quorum Management**: Always formally remove nodes intended for long-term maintenance to prevent API proxy instabilities.
