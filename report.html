Traefik IngressRoute Multiple Routes Analysis Report

Executive Summary

This report analyzes the current issue with qBittorrent routing through Traefik IngressRoute in a Kubernetes cluster, where multiple routes are configured on the same host (portal.local) but the /qbit PathPrefix route is not matching properly despite correct configuration.

Current Issue Analysis

Problem Description

The qBittorrent service is not accessible via http://portal.local/qbit despite proper IngressRoute configuration. The route returns 404 errors from Traefik itself, indicating the route is not being matched.

Current Configuration

# qbittorrent-ingress-route.yaml apiVersion: traefik.io/v1alpha1 kind: IngressRoute metadata: name: qbittorrent-ingress-route namespace: arr spec: entryPoints: - web - websecure routes: - match: Host(`qbittorrent.local`) || Host(`qbittorrent.pindaroli.org`) kind: Rule services: - name: servarr-qbittorrent-web port: 8080 - match: "Host(`portal.local`) && PathPrefix(`/qbit`)" kind: Rule middlewares: - name: qbit-stripprefix services: - name: servarr-qbittorrent-web port: 8080 tls: {}

Conflicting Configuration

Route Conflict Detected: The radarr-external IngressRoute also uses Host(`portal.local`): 
match: Host(`portal.local`) && PathPrefix(`/radarr`) || Host(`radarr.pindaroli.org`)
This configuration has operator precedence issues that may affect routing.

Research Findings

1. Traefik IngressRoute Multiple Routes Issues

  • Route Priority: Routes are sorted by default in descending order using rule length. Longer rules get higher priority.
  • Path vs PathPrefix: PathPrefix matches the path and anything that starts with that path.
  • Service Namespace Restrictions: You can only expose services in the same namespace as your IngressRoute resource.
  • Operator Precedence: Complex match expressions with multiple operators need careful consideration.

2. PathPrefix Matching Problems

Common Issues Found:
  • Trailing slash importance in PathPrefix rules
  • Path forwarding behavior - backend must expect the forwarded path
  • Route conflicts when multiple IngressRoutes use the same host
  • Middleware execution only when route rules match

3. Route Conflicts in Same IngressRoute

  • Combined Routes: More efficient for related services under the same domain
  • Separate Routes: Better isolation and security boundaries
  • TLS Conflicts: Same host with different TLS options can cause conflicts
  • Entry Point Mismatches: Can cause routing errors

Root Cause Analysis

Primary Issue: Operator Precedence in Radarr Route

The radarr-external IngressRoute contains a problematic match expression:

Host(`portal.local`) && PathPrefix(`/radarr`) || Host(`radarr.pindaroli.org`)

Problem: This expression is interpreted as:

(Host(`portal.local`) && PathPrefix(`/radarr`)) || Host(`radarr.pindaroli.org`)

However, due to Traefik's route processing, this might interfere with other routes on the same host.

Secondary Issues

  • Route Length Priority: The radarr route is longer and gets higher default priority
  • Configuration Reload: Traefik may not have properly reloaded the updated configuration
  • Middleware Dependencies: The qbit-stripprefix middleware requires the route to match first

Recommended Solutions

Solution 1: Fix Operator Precedence (Immediate)

Recommended: Fix the radarr route to use proper parentheses:
# In radarr-external IngressRoute match: (Host(`portal.local`) && PathPrefix(`/radarr`)) || Host(`radarr.pindaroli.org`)

Solution 2: Add Explicit Priorities (Alternative)

# qBittorrent route with high priority - match: "Host(`portal.local`) && PathPrefix(`/qbit`)" kind: Rule priority: 1000 middlewares: - name: qbit-stripprefix services: - name: servarr-qbittorrent-web port: 8080

Solution 3: Separate IngressRoutes (Long-term)

Best Practice: Create separate IngressRoute resources for each service:
  • qbittorrent-portal-ingress-route.yaml - for portal.local/qbit
  • radarr-portal-ingress-route.yaml - for portal.local/radarr
  • Keep existing direct domain routes separate

Solution 4: Debugging Steps

Immediate Actions:
  1. Check Traefik dashboard at port 8080 for active routes
  2. Verify middleware is loaded: kubectl get middleware -n arr
  3. Test with explicit priorities before restructuring
  4. Monitor Traefik logs for routing errors

Best Practices for Future Configurations

Route Design Guidelines

  • Use explicit priorities for complex routing scenarios
  • Separate concerns - different applications should have separate IngressRoutes
  • Consistent naming - use clear, descriptive names for routes and middlewares
  • Namespace isolation - keep services and IngressRoutes in the same namespace

Debugging Tools and Techniques

  • Use Traefik dashboard for visual route inspection
  • Check pod logs for configuration errors
  • Test with curl to avoid browser caching issues
  • Verify service endpoints are healthy before troubleshooting routes

Security Considerations

  • Avoid wildcard routes that could be hijacked
  • Use proper TLS configuration for sensitive services
  • Consider using separate entry points for internal/external traffic
  • Implement proper authentication middleware where needed

Conclusion

The qBittorrent routing issue is primarily caused by operator precedence problems in the radarr route configuration, combined with Traefik's default route priority system. The immediate fix is to correct the operator precedence in the conflicting route, while the long-term solution involves restructuring routes for better isolation and maintainability.
